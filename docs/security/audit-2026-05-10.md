## Scope Change

The Feb 2026 audit profile said *"File handling: Server-side image downloads from Wikimedia/UNESCO (no user uploads currently)"* and listed *"V5 user file uploads"* as **out of scope**. Both statements were stale: cv-python's `/pipeline/phase1`, `/pipeline/phase2`, and `/pipeline/match` endpoints accept `UploadFile` parameters (multipart) carrying curator-supplied map images. V5 was therefore **moved into scope** in this audit.

A new **Trust Boundaries** section was added to `SECURITY.md` to make the Node↔cv-python boundary explicit:

- Browser ↔ Node backend: public, fully hardened.
- Node backend ↔ cv-python: private Docker bridge, no per-request authentication. Trust derives from network isolation — cv-python has no `ports:` mapping in `docker-compose.yml` (only `expose: 8000`), so it's reachable only at `http://cv-python:8000` from other bridge-resident containers. ASVS L2 considers this defence-in-depth — adequate for the current threat model — but if cv-python ever moves out of the bridge a shared-secret header or mTLS becomes mandatory.

## Findings & Remediations

### Pre-audit findings (cv-python)

| # | Severity | Area | Finding | Remediation |
|---|---|---|---|---|
| 1 | HIGH | V4.1.5 / V5.1.1 | No request-body size limit. A multi-GB upload could exhaust memory. | Added `BodySizeLimitMiddleware` (`cv-python/app/middleware.py`) — caps Content-Length at `CV_MAX_BODY_BYTES` (default 100 MB), 413 response when exceeded. |
| 2 | MEDIUM | V5.1.2 | `decode_image()` returned `cv2.imdecode`'s output directly; the function returns `None` for non-image bytes, so callers fed garbage saw a worker crash with full traceback rather than a clean 4xx. | `decode_image()` now raises `InvalidImageBytes` on empty bytes / `None` return; FastAPI exception handler maps it to a 400 with a generic detail string. |
| 3 | MEDIUM | V13.4 | The NDJSON streaming worker emitted `{"type":"error","message": str(e)}` — raw exception text leaks paths, stack frames, internal state. | Worker now logs the full traceback to stderr (Docker stdout) and emits a generic `"Internal error during processing"`. Top-level FastAPI exception handler does the same for non-streaming routes. |
| 4 | MEDIUM | V4.1.5 | Each request spawned a `threading.Thread(daemon=True)` with no upstream cap. 100 simultaneous calls = 100 in-flight image decodes. | Dockerfile CMD now passes `--limit-concurrency 8` to uvicorn. Excess connections receive a 503 instead of stacking workers. |
| 5 | LOW | V2.4.1 | `params: str = Form(...)` then `json.loads(params)` is unvalidated; malformed JSON triggers a generic 500. | Documented as a follow-up (migrate to Pydantic models). The error path is now safe (no info leak) so this is no longer blocking. |
| 6 | LOW | V8.1.1 | cv-python has no per-request auth; trust derives from Docker network isolation. | Documented as a trust boundary in `SECURITY.md`. Tracked as a follow-up (shared-secret header or mTLS) for any deployment that broadens the boundary. |

### Verified clean

- **V1.2.5 (OS command injection)**: cv-python source contains no `subprocess.*` calls — verified by code review.
- **V11/V12 (crypto, TLS)**: cv-python performs no crypto operations and makes no outbound network calls.
- **V13 (config / containers)**: Dockerfile runs as non-root `appuser`, uses pinned `python:3.12-slim`, no secrets baked in. Trivy is configured to fail CI on HIGH/CRITICAL CVEs in the image.
- **Bandit findings**: 3 pre-existing `try/except/pass` warnings (B110) in `cv/match.py` are intentional projection-search exception handlers. Suppressed inline with `# noqa: S110  # nosec B110 -- <reason>` per the suppression policy.

## Open follow-ups (non-blocking)

1. **cv-python params validation** — migrate `params: str = Form(...)` + `json.loads()` to first-class Pydantic models so schema violations land at the FastAPI layer, not the worker.
2. **cv-python structured logging** — replace `print()` with the standard `logging` module + JSON formatter so security-relevant events (when added) flow into a single log stream.
3. **cv-python inter-service auth** — add a shared-secret `Authorization` header check via `Depends` for any deployment that exposes cv-python beyond the Docker bridge.
4. **mypy tightening on `app.cv.*`** — currently silenced via `follow_imports=skip`+`ignore_errors=true` because numpy/cv2 stubs produce ~150 spurious findings. Re-enable file-by-file as type annotations land.
5. **Bandit severity floor** — currently default (Low+). Consider raising to medium once the codebase is annotation-rich and the noise floor is understood.
